SquidRouterModule Safe Wallet Vulnerability Investigation Report

Investigation Date: May 26, 2026
Event Type: Safe Wallet Module Vulnerability Exploitation
Loss Amount: $3,200,000
Affected Wallets: 86 Gnosis Safes
Attack Chain: Ethereum + Base

📋 Event Overview

On May 26, 2026, blockchain security company Blockaid detected a sustained attack targeting the SquidRouterModule contract. Attackers stole a total of $3.2 million in crypto assets from 86 Gnosis Safe wallets within approximately 2 hours.

Funds were converted to DAI via attacker-controlled Uniswap V3 pools and then consolidated to a single wallet address. PeckShield confirmed the attacker initially received 2.1 ETH from Tornado Cash as seed capital.

Attacker Wallet: 0xA447...54859

🔍 Attack Method Analysis

Attack Flow

  1. Vulnerability Identification: Attackers discovered a design flaw in the executeSameChainActions() function of the SquidRouterModule contract
  2. Authorization Acquisition: Victim Safe wallets added the malicious contract as a "Trusted Safe Module"
  3. Privilege Abuse: This module could control arbitrary tokens in the Safe wallet without signatures
  4. Exchange & Laundering: Stolen tokens were swapped via Uniswap V3 for nearly worthless malicious token "u"
  5. Fund Consolidation: All valuable assets were ultimately converted to DAI and consolidated to attacker wallet

Core Vulnerability

According to the official Squid statement:

"The root cause of the vulnerability was a third-party module incorrectly assuming that a publicly visible constant string was sufficient to indicate 'safe'. If you passed in this string (which is publicly available), you could execute arbitrary call data and steal arbitrary funds."

Victim Behavior

Victim Safe wallets added SquidRouterModule as a "Trusted Safe Module." This authorization was intended to allow the contract to execute certain operations on behalf of the Safe, but the contract's design flaw made it exploitable.

⚠️ Relationship with Squid Protocol

Official Squid Statement

⚠️ This is NOT a Squid Protocol security incident

"This is a third-party SquidRouterModule being exploited, not our protocol's Router contract."

"The affected contracts used our name, but not our code."

Safe Labs Confirmation

Safe Labs CEO Rahul Rumalla stated: - Preliminary investigation indicates affected accounts were not operated through the official Safe Wallet product - The malicious module had previously been flagged by Blockaid and added to the Safe Shield risk detection framework

🛡️ Community Protection Recommendations

For Safe Users

  1. Module Authorization Caution: Carefully review any third-party Safe modules
  2. Verify Contract Source: Ensure modules come from trusted developers
  3. Check Module Permissions: Understand what modules can do once authorized
  4. Use Safe Shield: Enable protection provided by security services like Blockaid

For Protocol Developers

  1. Security Assumptions: Don't assume publicly visible strings or constants are "proof of safety"
  2. Principle of Least Privilege: Grant only necessary permissions
  3. Security Audits: Third-party integrations should undergo independent audits

📝 Unique Analysis Perspective

Third-Party Module Risks

This attack reveals systemic risks of "third-party dependencies" in the DeFi ecosystem:

  • Name Confusion: Malicious contracts use legitimate project names, causing user misjudgment
  • Trust Transfer: Users trust Safe wallet, but modules trusted by Safe may not be trustworthy
  • Responsibility Vacuum: When third-party modules are exploited, responsibility attribution is unclear

Gnosis Safe's "Trusted Safe Module" Mechanism

Safe wallet's "Trusted Safe Module" mechanism has good intentions: - Allows authorized contracts to execute operations on behalf of the wallet - Supports automated strategies and cross-chain functionality - Improves usability

But it also brings risks: - Module gains full control - Once abused, consequences are severe - Users have difficulty assessing module security
