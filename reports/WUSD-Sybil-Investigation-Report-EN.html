I. Executive Summary

On May 25, 2026, the WUSD.fi protocol suffered a sophisticated Sybil farming attack. Attackers exploited a flaw in the protocol's reward mechanism design, using EIP-7702 technology to batch-create wallet addresses for farming GLOVE token rewards, ultimately stealing approximately $207,000 from Uniswap V3 liquidity pools.

Key Findings: - This is the first large-scale attack using EIP-7702, marking the maturation of a new attack technique - The core vulnerability was WUSD._englove() function lacking Sybil resistance mechanisms - Attackers converted funds to 98 ETH and moved to Railgun privacy protocol, increasing tracing difficulty - As of report date, WUSD.fi team has not issued an official statement

III. Vulnerability Analysis

3.1 Vulnerability Mechanism: WUSD._englove() Design Flaw

Vulnerable Code Logic:

Condition 1: Wallet is fresh (new wallet)
Condition 2: Wrap ≥ 100 WUSD
Condition 3: Holdings < 2 GLOVE
→ Can call Glove.mintCreditless() to receive 2 GLOVE

Triple缺失: | Protection Measure | Status | |-------------------|--------| | Identity Verification | ❌ Missing | | Rate Limiting | ❌ Missing | | Address Correlation Detection (Sybil Detection) | ❌ Missing |

Attack Feasibility: - Any new wallet address meeting conditions can claim 2 GLOVE tokens - Attackers can batch create addresses to infinitely farm rewards - Contract code logic is completely correct, but economic incentive design has fundamental flaws

3.2 EIP-7702 Attack Mechanism Deep Analysis

What is EIP-7702: EIP-7702 is a new feature introduced in Ethereum's Pectra upgrade, allowing externally owned accounts (EOAs) to temporarily delegate execution authority to smart contracts, enabling regular wallets to operate as contracts.

Key Role in Attack:

Traditional method: Creating each new wallet requires:
         1. Generate private key
         2. Deploy wallet contract (or use EOA)
         3. Transfer funds
         4. Contract calls
         → High cost, low efficiency per operation

EIP-7702 method:
         1. Deploy single helper contract
         2. Batch-proxied multiple EOA addresses via EIP-7702
         3. Contract internally automates all operations
         → Dramatically reduces batch operation cost, enables scaled attacks

Technical Breakthrough: - Attacker only needed to deploy one EIP-7702 helper contract - This contract could proxy unlimited EOA addresses to execute smart contract logic - Each proxied address appears as "fresh wallet" to the protocol - Achieved single contract, multiple addresses, large-scale Sybil farming attack

EIP-7702 Security Warning: This is another case of EIP-7702 being used for malicious purposes since the Pectra upgrade in May 2025. Previous phishing attacks using this technology resulted in $1.54M in losses.

8.1 Vulnerability Root Cause Summary

Dimension Problem
Code Level Contract logic correct, no typical vulnerabilities
Design Level WUSD._englove() lacks Sybil resistance
Economic Level mintCreditless has no rate limit/identity verification
Audit Level Regular audits don't test economic incentive paths

8.2 Protocol Security Recommendations

Immediate Actions: 1. Pause Glove.mintCreditless() functionality 2. Implement wallet history correlation detection 3. Add single address claim frequency limits 4. Introduce on-chain identity verification (e.g., WorldID)

Long-term Improvements: 1. Economic incentive design needs specialized audits 2. Introduce TWAP price oracle to prevent flash loan manipulation 3. Establish real-time anomaly monitoring alert system 4. Consider decentralized emergency pause mechanism
